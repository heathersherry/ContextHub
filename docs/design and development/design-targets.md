## ContextHub 设计与功能概览

---

### 一、背景：多 AI Agent 协作为什么会出问题？

当企业同时部署多个 AI Agent 协作完成业务流程时（比如一个负责数据分析、一个负责客户沟通、一个负责合规审查），各个 Agent 各自记录自己的"工作记忆"，互相看不到对方的进展。研究显示，**多 Agent 系统在生产环境中有 36.9% 的失败率[[Cemri et al., 2025](https://arxiv.org/abs/2503.13657)]，其中 79% 根因是协调问题[[Zylos Research, 2026](https://zylos.ai/research/2026-03-09-multi-agent-memory-architectures-shared-isolated-hierarchical)]，而不是 AI 本身不够聪明**。

具体问题有五类：

- **信息孤岛**：A Agent 做完了客户意向评估，B Agent 看不到，还在给同一个客户发通用邮件
- **策略不同步**：法务部门更新了合规政策，但各业务 Agent 各自存了一份旧版本，无法自动同步
- **记忆不结构化**：自由文本的记忆无法按阶段过滤或与 CRM 系统对齐
- **冗余注入**：每个任务步骤都要重复塞入相同的背景知识，浪费大量 token
- **无反馈的静默退化**：Agent 依赖的数据表字段悄悄改了，系统几个月后才发现

**ContextHub** 就是为了解决这些问题而设计的——一个让多个 AI Agent 共享、管理、同步"上下文"的中间件（是否是企业级？有待各位讨论出“企业级”的定义）。

### 二、底层基础："一切皆文件"的接口+数据库存储后端

#### 2.1 统一 `ctx://` URI 路由

**通俗解释：** 每一块上下文（无论是 AI 的工作记忆、技能指令、文档资料还是数据表结构）都有一个统一格式的"地址"，就像网页有 URL 一样。Agent 不需要知道数据存在哪台机器、哪个数据库，只要拿着这个地址就能读写。

```
ctx://team/skills/sql-generator         → 全组织共享的 SQL 生成技能
ctx://team/engineering/memories/        → 工程部的共享记忆
ctx://agent/bot-001/memories/cases/     → 某个 Agent 的私有案例库
ctx://datalake/hive/prod/orders         → 数据湖里的 orders 表元数据
```

**好处：** 对 Agent 来说，操作上下文就像操作文件系统（`ls` 浏览目录、`read` 读内容、`grep` 搜索），直觉清晰；而且不管内部存储怎么变，这个"地址格式"永远不变，Agent 写的代码不需要改。


#### 2.2 单数据库架构（PostgreSQL 为核心）
```
        用户
         │
    ┌────┴─────┐
    │ OpenClaw │  ← Agent 运行时
    └────┬─────┘
         │
    ┌────┴─────────┐
    │ ContextHub   │  ← 上下文管理中间件（应用层）
    └────┬─────────┘
         │
    ┌────┴───┐
    │  PG    │  ← 存储层
    └────────┘
```

**通俗解释：** 所有数据（元数据、内容、向量索引、权限记录、变更事件）都存在同一个 PostgreSQL 数据库里。没有额外的向量数据库、没有消息队列中间件、没有独立的文件存储系统（小文件）。

**为什么不用文件系统或多套系统？**


| 需要的能力  | 单 PG 怎么做              | 多系统方案的代价              |
| ------ | --------------------- | --------------------- |
| 变更通知   | PG 原生 `LISTEN/NOTIFY` | 自建消息队列，维护成本高          |
| 依赖关系查询 | SQL JOIN + 递归查询       | 解析文件、遍历目录             |
| 事务一致性  | ACID 保证               | 写内容和写元数据可能不一致         |
| 权限隔离   | 行级安全策略（RLS）           | 应用层路径检查，容易漏洞          |
| 向量语义检索 | pgvector 扩展，同库同事务     | 双写一致性问题（PG 写成功但向量库失败） |


* **好处：** 架构极简，可以私有化部署在企业内网（区别于大多数竞品只提供 SaaS）；事务保证内容和元数据永远一致；一套数据库就能搞定检索、权限、版本、传播所有事情。
* **参考：** db9项目(Postgres but for agents) [https://db9.ai/]

### 三、信息分层：按需喂给 AI，不浪费 token

#### L0 / L1 / L2 三层信息模型

**通俗解释：** 每条上下文都有三个"精度档"，就像地图有全国视图、城市视图、街道视图。Agent 先看最简版本，觉得相关再往下看详细版，不需要的精度一律不传。


| 层级        | 大小      | 用途            | 举例（以 orders 表为例）                   |
| --------- | ------- | ------------- | ---------------------------------- |
| **L0 摘要** | ~100 词  | 语义检索、快速判断是否相关 | "orders 表 - 存储所有订单交易记录，包含金额、状态、时间" |
| **L1 概览** | ~2000 词 | 精排、内容导航       | 字段列表 + 类型说明 + 样例数据                 |
| **L2 详情** | 不限      | 按需加载，真正用到时才取  | 完整 DDL + 血缘 + 历史查询模板               |


**好处：** 只有 L0 摘要会被向量化用于语义检索，L1/L2 只在命中后按需读取。一次检索的 token 消耗可以只有全文加载方案的 1/10 以下。同时 L0 向量索引和内容存在同一个数据库里，不存在"向量库找到了但内容库没有"的不一致问题。

### 四、多租户 + 团队层级：谁能看什么

#### 4.1 RLS 行级安全隔离（多租户）

**通俗解释：** 一套 ContextHub 系统可以同时服务多家企业客户（租户）。每条数据库记录都打上了"属于哪家公司"的标签，数据库层面直接保证 A 公司的 Agent 永远看不到 B 公司的任何内容，不靠应用层的代码去过滤（那容易出 bug）。

**好处：** 权限在数据库引擎层强制执行，即使应用代码有漏洞，数据也不会跨租户泄露。每个数据库连接在建立时就锁定了"我属于哪个租户"，之后所有查询自动带上这个过滤条件。

#### 4.2 多层级团队所有权模型

**通俗解释：** 内容的归属遵循企业组织架构。每条上下文都属于某个"层级"，Agent 能看到自己所在团队以及所有上级团队的内容，但**看不到下属子团队的私密内容**（上级不默认可见下属的工作细节）。

```
全组织（根团队）
  ├── 工程部
  │   ├── 后端组         ← Agent A 的位置
  │   └── 数据组
  └── 销售部

Agent A 可以看到：
  ✅ 自己的私有空间
  ✅ 后端组的共享内容
  ✅ 工程部的共享内容
  ✅ 全组织根团队的共享内容
  ✅ 数据湖表（默认全组织可见）
  ❌ 销售部的私密内容（跨团队，不可见）
```

**好处：** 遵循"最小权限"原则，既不让信息孤岛，也不让所有人都能看所有东西。组织层级自动决定可见范围，不需要管理员手动给每个 Agent 配权限。

#### 4.3 显式 ACL 访问策略（后续版本）

**通俗解释：** 在默认的团队层级可见性之上，管理员可以设置"谁能读什么"的精细规则，而且**"禁止"永远比"允许"优先**。比如：全公司默认能看数据湖表，但可以专门设一条规则"只有 HR 团队能看薪资表，其他人的访问请求一律拒绝"。

规则支持：

- **路径通配**：`ctx://datalake/prod/salary/`* 可以一次性限制某个路径下的所有内容
- **字段脱敏**：允许读取一张表，但敏感字段（比如身份证号）自动被替换为 `[MASKED]`，Agent 看到的是脱敏版
- **deny 优先**：父团队的禁止规则不能被子团队的允许规则覆盖（不允许下级绕过上级的安全策略）

**好处：** 默认走团队层级自动判断，不需要为每个资源都写规则；只有"例外情况"（某人不该看某个特定资源）才需要显式 ACL 介入，管理成本低。

### 五、版本管理：Skill 的"不可撤销发布"与破坏性变更

**通俗解释：** "Skill"是 Agent 的技能指令（比如"如何生成 SQL"的一段 Markdown 提示词）。和软件一样，Skill 会迭代更新。ContextHub 管理这个版本历史：

- **一旦发布就不可修改**：已发布的版本内容永久保留，可以随时回溯，确保任何人事后还能看到"当时用的是什么指令生成的结果"
- **发布者手动标记"破坏性变更"**：如果新版 Skill 改了用法，发布者需要打一个 `is_breaking=true` 的标记
- **两种订阅模式**：
  - *浮动订阅*：始终跟随最新版本，有新版本时收到通知
  - *固定版本*：锁定在某个稳定版本，新版本发布时收到"提示"（"v3 已出，你还在 v2"），但不被强制切换

**好处：** Skill 的发布者和使用者解耦。一个团队可以在稳定版本上安全运行，另一个团队可以先试用新版，互不干扰。同时历史版本永久可查，便于排查问题（"当时为什么生成了这样的 SQL？"）。

### 六、自动变更传播：上游改了，下游自动知道

**通俗解释：** 上下文之间有依赖关系。比如一个 Agent 的工作案例是用 Skill v2 生成的，那这个案例就"依赖"Skill v2。当 Skill 发布了 v3（破坏性变更），系统会自动找到所有依赖它的内容，并采取相应动作——而不是等人发现"这个结果是用旧版本生成的"。

**三级响应策略，尽量不烧 token：**


| 级别            | 适用场景                        | 做法                    | token 消耗   |
| ------------- | --------------------------- | --------------------- | ---------- |
| **级别 1：纯规则**  | Skill 打了 `is_breaking=true` | 自动把依赖的案例标记为"过时"       | 0          |
| **级别 2：模板替换** | 共享记忆被纠正了一个事实性描述             | 找到对应文字，直接字符串替换        | 0          |
| **级别 3：模型推理** | 复杂变更，需要判断"是否真的影响"           | 给 AI 看变更摘要和上下文摘要，让它判断 | ~230 token |


**实际效果：** 一次 Skill 升级影响 10 个依赖方，若用纯规则处理，总消耗约 230 token；若全部走 AI 推理，则需约 20,000 token——**节省 99%**。

传播的可靠性：依赖关系存在数据库里，变更事件也持久化，即使系统崩溃重启，传播引擎也会从数据库重新扫描"未处理的事件"，不会漏掉。

**内容的冷热分层：** 被标记为"过时"（stale）的内容不会立刻删除，仍可通过地址直接读取，但在语义检索结果中权重会降低。超过一定时间无人访问后，会自动进入"归档"状态——从向量索引中退出（不再出现在检索结果里），但数据行保留，仍可按地址精确访问。再往后，可选择彻底删除。这形成了 `active → stale → archived → deleted` 的完整生命周期，冷数据不占用检索资源，也不会被误删。

### 七、记忆晋升机制：私有经验变成团队知识

**通俗解释：** Agent 在日常工作中积累的私有经验（比如"发现 orders 表用这个 SQL 模板效果最好"），可以被"晋升"为团队共享知识，让整个工程部的 Agent 都能受益。晋升后，系统会记录"这条知识是从哪里来的"，一旦原始记忆被修改，下游派生的共享记忆也会收到通知。

```
Agent私有记忆
  ctx://agent/backend-bot/memories/cases/sql-pattern-001
      ↓  晋升
团队共享记忆
  ctx://team/engineering/memories/shared_knowledge/sql-pattern-001
      ↓  工程部下所有子团队 Agent 可见
      ↓  dependencies 记录 derived_from，源记忆变更时传播通知
```

**好处：** 把一个 Agent 踩过的坑和积累的最佳实践，沉淀为整个团队的组织级知识，不会因为某个 Agent 下线而丢失。同时来源可追溯，便于审计。

### 八、数据湖表管理（如果需要的话可添加这个验证场景，这是为了Data Agent设计）

**通俗解释：** 企业数据湖里有成百上千张表，Agent 做数据分析前需要先"理解"这些表——字段是什么、数据量多大、表之间如何关联、适合用什么 SQL 模板。ContextHub 把每张表的元数据也纳入统一的 `ctx://datalake/...` 体系管理。

关键设计：表的元数据**按更新频率分拆存储**——

- 表结构（DDL）：很少改，改了触发变更传播
- 分区信息：偶尔更新，不触发传播
- 行数/大小等统计：每次数据同步都更新，**不触发传播**（否则每次刷新数据都会骚扰所有 Agent）

Agent 查询数据时，拿着自然语言问题，先向量搜索找到最相关的几张表（看 L0 摘要），再按需加载字段说明（L1）和完整 DDL + 查询模板（L2），组装成 Text-to-SQL 所需的上下文。

**好处：** 数据湖表元数据和 AI 记忆/技能用同一套体系管理，Agent 做 SQL 任务不需要知道表在哪个系统里，直接通过 `ctx://` 地址访问；表结构变更时自动传播通知，不再有"表改了但 Agent 还在用旧 schema 生成 SQL"的静默失败。

### 九、长文档检索

**通俗解释：** 对于动辄几百页的财报、法规文件、技术手册，传统的"切段+向量检索"方式（把文档切成小块，向量检索最相关的块）会截断语义边界，找到的答案容易不准确。ContextHub 计划引入两种更先进的检索策略：

- **树导航检索**：按文档自身的章节结构建一棵目录树，AI 像翻目录一样逐层推理定位到精确章节，再读取原文。（类似于你找一本书里的某段话，先看目录定位到章节，再翻到那一页。）准确率可达 95% 以上，远优于传统切块方式的 75–80%。
- **关键词+随机采样检索**：用 ripgrep 在原文件上直接搜关键词，找到命中位置后用概率采样的方式扩展上下文窗口，提取证据。速度极快（0.5–1 秒），适合快速筛选。

大文档的原文存在文件系统里（用于快速搜索），目录树结构存在数据库里（用于 AI 推理定位），两者协作，各司其职。

**好处：** 解决了企业场景中"精确引文"需求——不是笼统地说"文档里提到过"，而是能精确定位到"第三章第二节第四段说了这句话"，满足合规、审计等对精准度要求极高的场景。

### 十、内容生命周期与质量反馈

#### 10.1 生命周期管理（已内置于核心设计）

**通俗解释：** 系统里每一条上下文都有自己的"健康状态"，并随时间和使用情况自动流转：

```
创建 ──→ active（正常使用）
              │
        长期未访问，或被变更传播标记为过时
              ↓
           stale（过时，检索降权，但仍可直接读取）
              │
        持续无人访问超过一定天数
              ↓
          archived（归档，退出向量索引，不再出现在检索结果中）
              │
        （可选）超时后
              ↓
           deleted（逻辑删除）
```

生命周期策略可按内容类型和归属层级分别配置——例如 Agent 私有记忆 90 天未访问自动降级，团队级技能规则宽松一些，数据湖表元数据则永不自动归档（由数据湖同步任务主动触发）。

**好处：** 系统自我维护，不需要人工定期清理"僵尸"上下文。活跃内容始终在检索结果的前排，冷数据安静退场但不丢失，随时可以按地址召回。

#### 10.2 质量反馈闭环（后续版本）

**通俗解释：** 当前系统是"单向的"——把上下文喂给 Agent，但不知道 Agent 有没有真的用上。未来计划通过观察 Agent 的后续行为来推断每条上下文的实际价值：

- **采纳（adopted）**：Agent 在生成结果中确实用到了这条上下文
- **忽略（ignored）**：上下文被检索返回，但 Agent 完全没有用它
- **纠正（corrected）**：Agent 使用了，但用户事后纠正了结果（说明上下文可能有误）

这些信号会回写到每条上下文的热度评分里——采纳率高的内容检索排名自动上升，长期被忽略的内容排名下降，形成"有用的越来越容易被找到"的正循环。

**好处：** 检索质量随使用积累自动优化，不需要人工标注哪些内容"好"哪些"不好"；同时也能定期生成"低质量上下文报告"，帮助管理员发现并清理长期被忽略的噪音内容。

### 整体设计原则

1. **对外简单，对内强大**：Agent 看到的是熟悉的文件系统操作（读/写/搜索），内部是 PostgreSQL 提供的事务、权限、版本、事件通知等一切企业级能力。
2. **尽量不烧 token，必要时才用 AI**：变更传播的 70% 场景靠纯规则处理，20% 靠字符串替换，只有 10% 真正复杂的情况才请 AI 推理——在保证智能性的同时，把运营成本降到最低。
